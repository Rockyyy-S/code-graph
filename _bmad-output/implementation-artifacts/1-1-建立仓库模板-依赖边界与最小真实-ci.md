# Story 1.1: 建立仓库模板、依赖边界与最小真实 CI

Status: review

<!-- Note: 本 Story 已按 Create Story checklist 完成上下文分析；实现完成状态仍由 dev-story 与后续 code-review 决定。 -->

## Story

As a 项目贡献者，
I want 获得结构清晰、命令可重复执行的项目仓库，
so that 后续切片可以在明确的模块责任和依赖方向上独立开发。

## Acceptance Criteria

1. **Given** 一个新的项目检出  
   **When** 安装锁定版本的 Node.js、pnpm 和项目依赖  
   **Then** 仓库建立 `apps/*`、`packages/*` 和 `packages/adapters/*` 结构  
   **And** `apps/extension` 来源于官方 VS Code TypeScript + esbuild 模板  
   **And** 根级 `type`、`lint`、`unit`、`build`、`contract`、`dependency-boundary` 和 `basic-security` 命令可重复执行。

2. **Given** 核心、应用、契约和适配器包已经建立  
   **When** 执行依赖边界检查  
   **Then** domain 不依赖其他项目包  
   **And** application 只向 domain 和自身稳定接口依赖  
   **And** 适配器不能被核心反向导入  
   **And** graph-service 被保留为唯一组合根。

3. **Given** 贡献者需要放置共享代码  
   **When** 选择目标包  
   **Then** 代码按领域、应用、契约、客户端或适配器责任归位  
   **And** 仓库中不存在无责任边界的通用 utils 包  
   **And** 新包违反约定时依赖边界检查给出相对路径和修复建议。

4. **Given** 根级 `type`、`lint`、`unit`、`build`、`contract`、`dependency-boundary` 和 `basic-security` 命令已经可重复执行  
   **When** Pull Request 或受保护分支触发 CI  
   **Then** `architecture-required` 以稳定 check 名执行所有命令  
   **And** 任一真实失败阻止 Story 1.1 合并  
   **And** 不使用空测试、永久 skip、无断言或始终成功脚本。

5. **Given** `architecture-required` 的最小 CI 已配置  
   **When** 审查 Story 1.1 的独立完成证据  
   **Then** 该 required check 对当前及后续 Pull Request always-run，且不使用 path filter 跳过执行  
   **And** 受控失败证据证明任一真实失败能够阻止合并  
   **And** 完成证据不依赖 Story 1.2 已开始、提交或合并。

### 架构与依赖边界

| 单元 | 允许依赖 | 禁止事项 |
| --- | --- | --- |
| `packages/domain` | Node/标准库中与领域无关的最小能力；原则上无项目包依赖 | application、contracts、service-client、adapter、VS Code、SQL、IPC DTO |
| `packages/application` | `packages/domain`；在本包内定义稳定端口 | adapters、VS Code、SQLite、TypeScript Compiler API、传输 DTO |
| `packages/contracts` | 独立的共享 Schema/DTO 基础 | 领域行为、适配器实现、渲染库内部格式 |
| `packages/service-client` | contracts | 业务查询语义、图谱存储、adapter |
| `packages/adapters/*` | application/domain 所定义端口及各自技术依赖 | 被 domain/application 反向导入；承担组合根职责 |
| `apps/graph-service` | application、adapters、contracts | 把组合逻辑分散到其他 app；本 Story 提前实现握手/服务行为 |
| `apps/cli`、`apps/extension` | service-client、contracts | 直接访问 store/analyzer；复制索引或查询逻辑 |
| `apps/webview` | contracts | 直接连接 graph-service、读取文件或持有业务计算 |

- 主范式是六边形架构的模块化单体；依赖只向核心收敛，运行时最终只有一个本地图谱服务，不拆微服务。[Source: ../planning-artifacts/architecture/architecture-bmad-2026-07-13/ARCHITECTURE-SPINE.md#Design-Paradigm]
- `packages/application` 先保持单一 workspace；Structural Seed 中的 `indexing`、`cycles`、`querying`、`rules`、`impact`、`exporting`、`validation` 是该包内责任模块。未来若要拆为独立 workspace，必须先更新架构与边界合同。
- graph-service 是唯一组合根，但 Story 1.1 只建立目录、包元数据和边界规则；空服务、JSON-RPC 2.0、单实例、token/workspace-key/协议握手与 `ServiceStatusV1` 全部属于 Story 1.2。[Source: ../planning-artifacts/epics.md#Story-12启动空-graph-service-并完成协议握手]
- 禁止通用 `utils` 包。共享代码必须按 domain、application、contracts、service-client 或具体 adapter 的责任放置；无法归位时应先澄清 owner，而不是新建杂物包。[Source: ../planning-artifacts/architecture/architecture-bmad-2026-07-13/ARCHITECTURE-SPINE.md#Consistency-Conventions]

### 测试与完成护栏

- 最小 CI 必须真实运行 type、lint、unit、build、contract、dependency-boundary、basic-security，并在任一步失败时返回非零；`architecture-required` 自身不允许 path filter。[Source: ../planning-artifacts/architecture/architecture-bmad-2026-07-13/ARCHITECTURE-SPINE.md#AD-28]
- 七个门禁分别至少有一个可控失败测试，并有 provider/聚合层证据证明任一非零退出都会使 `architecture-required` 失败、阻止合并；仅验证“脚本存在”不等于验证其会阻断错误。
- 单元/合同测试必须包含断言；禁止 `.skip`、`.todo`、focused test、`passWithNoTests` 和空 fixture。
- 边界诊断必须使用相对 POSIX 路径，不泄露开发机绝对路径。
- Story 1.1 不承担 60s/300ms/2s 产品性能、平台 VSIX、Node runtime/SQLite ABI、真实 Webview、辅助技术或发布矩阵证据；不得在完成说明中宣称这些能力已达标。
