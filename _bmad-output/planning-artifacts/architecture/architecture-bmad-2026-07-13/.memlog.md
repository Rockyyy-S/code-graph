- (question) 主设计范式采用何种一致性模型：六边形架构的模块化单体、传统分层模块化单体，还是更强的多进程服务化分解？
- (decision) 采用六边形架构的模块化单体作为主设计范式：领域与应用核心定义输入/输出端口，VS Code、CLI、SQLite、Tree-sitter、LSP、Git 与渲染相关实现作为适配器向内依赖；该规则约束全部模块并防止基础设施类型、SQL、宿主 API 或渲染格式渗入核心。
- (decision) 索引子系统采用管道与过滤器模式组织采集变更、解析、规范化、计算图谱差量、持久化、派生视图、规则检查与状态发布；Worker 线程或子进程仅作为性能/故障隔离手段，不形成独立微服务。
- (question) 本地图谱服务的运行与状态所有权如何划分：每工作区一个可发现的常驻服务、每客户端各自启动服务，还是单个用户级多工作区守护进程？
- (decision) 每个工作区采用一个按需启动、可发现、可空闲退出的本地图谱服务实例；VS Code 与 CLI 通过本地 IPC 连接同一实例，Windows 使用命名管道，macOS/Linux 使用 Unix Domain Socket，应用协议为 JSON-RPC 2.0，不开放 TCP 监听。
- (decision) 本地图谱服务是工作区共享图谱、索引队列、schema 迁移与 freshness/stale 状态的唯一所有者和写入者；客户端只拥有选中节点、固定视图、筛选、缩放等会话 UI 状态。该规则防止重复索引、并发写冲突与客户端间状态漂移。
- (question) 图谱状态采用何种变更模型：分析器输出事实批次并由单一事务应用 GraphPatch、使用事件溯源重放全部变更，还是允许分析器直接 CRUD 持久层？
- (decision) 图谱状态采用 FactBatch → 规范化 → GraphPatch → 单事务提交的变更模型；分析器不得直接访问 GraphStore，只能输出带来源的观察事实。删除仅作用于同一工作区、输入切片与分析来源此前拥有的事实。
- (decision) 一次成功更新在同一事务中更新规范图谱和受影响 Findings，并递增 graphRevision；查询结果与 GraphViewModel 必须携带 revision。失败更新不可见，客户端继续使用上一 revision 并标记 stale。MVP 不采用完整事件溯源，只保留有界任务日志与 patch 摘要。
- (question) 节点、边与分析证据采用何种稳定身份：确定性工作区作用域 URI、持久随机 UUID，还是内容哈希；同一语义边的多分析来源是合并为证据集合还是保留为平行边？
- (decision) 节点与实体采用确定性的工作区作用域 cg:// URI；路径统一为工作区相对 POSIX 形式。Git 工作区键由规范化仓库身份与子路径的哈希生成，无稳定 Git 身份时由规范化本地 workspace URI 哈希生成。
- (decision) 规范语义边 ID 由 fromId、relationType、toId 确定；不同分析器结果作为 Evidence 附着到同一语义边，记录 provenance、confidence、sourceRange、analyzerVersion 与 lastSeenRevision。仅当所有有效 Evidence 消失时删除规范边。文件移动产生新路径 ID 并在 GraphPatch 中携带 movedFrom 映射。

- (question) MVP 运行时采用全 TypeScript/Node.js 单工具链、Rust 图谱服务加 TypeScript 客户端，还是从首版就采用 Node 编排加 Rust/WASM 热点的双工具链？
- (decision) MVP 采用全 TypeScript/Node.js 工具链：Node.js 24 LTS、TypeScript 7、pnpm 11 workspace；VS Code 扩展从官方 generator-code TypeScript + esbuild 模板起步。插件、CLI、graph-service 与共享契约使用同一语言。
- (decision) CPU 密集工作使用 worker_threads 或受控子进程隔离；只有基准证明确认某一适配器为持续瓶颈时，才在保持现有端口与 JSON-RPC 契约的前提下迁移到 Rust/WASM。首版不引入双工具链。
- (question) 仓库模块边界采用按能力划分的粗粒度六边形模块、传统技术分层目录，还是每个端口/实体独立包的细粒度模块？
- (decision) 仓库采用粗粒度能力模块加独立适配器：apps/graph-service、apps/cli、apps/extension、apps/webview；packages/domain、packages/application、packages/contracts、packages/service-client 与 packages/adapters/*。application 内按 indexing、querying、rules、impact、exporting 组织。
- (decision) domain 不依赖其他项目包；application 只依赖 domain 并定义端口；adapters 依赖 application/domain 并实现端口；graph-service 是唯一组合根；extension 与 CLI 只依赖 contracts/service-client，不导入领域、存储或分析器。contracts 只承载可序列化线协议 DTO。
- (question) TS/JS MVP 的分析权威如何划分：TypeScript Compiler/Language Service 主导并延后 Tree-sitter、Tree-sitter 主导且 TypeScript 只补充解析，还是两者从首版并行覆盖同类事实？
- (decision) TS/JS MVP 采用 TypeScript 7 Compiler API 作为 import/export、模块解析、tsconfig paths、project references、workspace 跨包引用与基础符号事实的唯一默认权威来源，并在 Worker 中维护增量 Program/Language Service。
- (decision) MVP 不依赖 VS Code 内置 TypeScript Server 私有状态，确保 CLI 与插件结果一致。Tree-sitter、LSP、SCIP 适配器延后；只有新增非 TS/JS 语言、TypeScript 分析无法达到性能目标或需要无有效 TS project 的轻量降级时才引入。
- (question) 可重建图谱、服务元数据与日志默认存放在操作系统用户缓存、仓库内 .codegraph 目录，还是采用用户缓存为默认且仅将规则/忽略配置留在仓库的混合策略？
- (decision) 采用混合本地数据布局：仓库只保存可共享的 .codegraph/rules.yaml 与 .codegraphignore；graph.sqlite、service.json、service.lock、有界日志和临时文件存入操作系统用户缓存下按 workspace-key 隔离的目录。
- (decision) 仅 graph-service 打开 SQLite；启用 WAL、foreign_keys、synchronous=NORMAL 与有界 busy_timeout，schema migration 事务化。损坏或迁移失败时保留时间戳故障副本并显式提示重建。日志不记录源码正文，默认使用相对路径并轮转。CLI 提供 cache path、cache clear 与 doctor。
- (question) 查询服务向 UI 返回预算内语义 GraphViewModel、原始图数据库行，还是已绑定具体渲染器坐标的预布局数据；预算、聚合、排序与截断由服务还是客户端负责？
- (decision) 查询服务返回预算内、渲染器无关的语义 GraphViewModel；服务负责查询、相关性排序、预算、聚合、截断、新增/既有风险、revision、freshness、completeness 与 confidence，Webview 负责渲染、布局和会话交互。
- (decision) GraphViewModel 不得暴露数据库行、SQL 概念、Cytoscape JSON 或像素坐标。初始查询返回完整模型；增量更新携带 baseRevision/nextRevision，客户端发现 revision 不连续时必须重新拉取完整模型。默认交互预算为 1 跳、100 节点、200 边，并由服务执行安全上限。

- (question) .codegraph/rules.yaml v1 的公共校验契约采用 JSON Schema 2020-12 + Ajv、Zod 运行时 schema 再导出 JSON Schema，还是手写 TypeScript 校验；YAML 诊断如何统一映射到行列位置？
- (decision) .codegraph/rules.yaml v1 采用 JSON Schema 2020-12 作为唯一公共契约，使用 yaml 2.9.0 保留 CST/范围并用 Ajv 8.20.0 严格校验；所有对象拒绝未知字段，version 必须为 1，三种规则按 type 判别。
- (decision) 配置诊断统一为包含 code、severity、message、relativePath、range、instancePath 与可选 ruleId 的 DTO，供 VS Code Problems、编辑器诊断、CLI 文本/JSON 与 doctor 复用。配置无效时不损坏图谱，上次规则 Findings 仅可标记 stale:config-invalid。
- (version) 2026-07-13 通过 Node.js v24.18.0 官方文档确认 node:sqlite 稳定级别为 1.2（Release candidate），Single Executable Applications 稳定级别为 1.1（Active development）；两者均未达到 Stability 2。
- (constraint) VS Code 官方发布文档支持按 target platform 构建平台特定扩展包，可用于携带平台相关原生依赖；运行时不得要求联网下载服务或依赖用户全局安装。
- (question) 服务与 CLI 如何发布：CLI npm + 平台特定 VSIX 各自携带同源服务构建、使用仍在活跃开发的 Node SEA 统一生成独立可执行文件，还是要求用户先全局安装外部 daemon？
- (decision) 交付采用 npm CLI + 平台特定 VSIX：CLI 要求 Node.js 24 LTS；VSIX 内含同源 graph-service 构建、启动恢复逻辑和目标平台/VS Code Electron ABI 对应的 better-sqlite3 原生模块，不在运行时联网下载或要求全局 daemon。
- (decision) 客户端与服务握手交换 clientVersion、serviceVersion、protocolVersion、schemaVersion；兼容则复用工作区服务，不兼容则拒绝并给出更新动作，禁止启动第二实例争抢工作区。MVP 正式目标为 Windows x64、macOS arm64/x64、Linux x64；Windows arm64 与 Linux arm64 延后。
- (question) VS Code 宿主表面如何分工：Activity Bar/TreeView + Webview + Problems 的混合原生方案、单一 Webview 应用，还是尽量只用原生 TreeView/列表而不提供主图 Webview？
- (decision) VS Code 采用混合宿主表面：Activity Bar/TreeView 负责状态入口与导航，Webview Editor 负责 Overview/Current Context/Changes 的图与等价列表，Problems 只承载可定位诊断，Status Bar 显示单行服务状态，全部操作同时注册 Command Palette。
- (decision) 扩展客户端拥有当前实体、固定状态、筛选、范围和视图模式等语义会话状态；Webview 只拥有缩放、滚动、布局位置等视觉状态。Webview 重载后由扩展重发完整 GraphViewModel 与语义状态。普通更新不弹通知，只有需用户恢复的失败才通知。
- (question) 索引、rebuild、check、impact 等操作采用显式后台 Job 协议、保持 JSON-RPC 长请求直到完成，还是由客户端轮询文件/数据库状态；并发、取消、重连和错误分类如何统一？
- (decision) 采用显式后台 Job 协议，状态为 queued/running/succeeded/failed/cancelled；每工作区只有一条图谱变更通道，查询并发读取最后已提交 revision。保存事件按路径合并并以 hash 去重，手动 rebuild 取消未执行增量任务并吸收其变化。
- (decision) Job 由服务持有并支持 job/start、job/get、job/cancel、进度与提交通知；副作用启动携带 clientRequestId 有限期去重。取消仅在安全点生效，SQLite 提交不可中断。check/impact/export 声明目标 revision。错误使用统一 category、retryable、相对路径、范围、logId 与恢复动作 DTO。
- (question) 不受信任工作区中是否完全禁止启动服务与索引、允许受限只读索引，还是保持完整功能；IPC、路径边界、Webview 网络和工作区文件解析采用何种安全护栏？
- (decision) 采用严格 Workspace Trust：VS Code 未受信任工作区不启动 graph-service、不扫描源码、不读取规则、不运行 Git 分析；信任撤销时停止 Job 并隐藏缓存结果。CLI 以用户显式调用作为读取授权，但不得执行项目脚本、TypeScript plugins、自定义 transformer 或可执行配置。
- (decision) IPC 与路径安全采用拒绝优先：POSIX 缓存目录 0700/socket 0600，Windows ACL 仅当前用户；连接校验随机令牌、workspace-key 与协议版本。所有路径 realpath 后验证位于 workspace roots，拒绝穿越和指向外部的符号链接。Webview 使用严格 CSP、nonce、本地资源根和消息 Schema 校验，不直接连接服务。
- (question) CLI 公共契约采用单一 codegraph 子命令体系、多个独立可执行文件，还是仅暴露底层 JSON-RPC；人类/机器输出、stdout/stderr 与退出码如何稳定定义？
- (decision) CLI 采用单一 codegraph 子命令体系：rebuild、query、check、impact、export、status、doctor、cache path/clear。默认文本输出，--format json 提供 schemaVersion:1 机器契约，impact/export 支持 markdown；正常数据 stdout，进度与可恢复警告 stderr。
- (decision) CLI 退出码固定为 0 成功或仅 warning、1 error 规则违反/失败条件命中、2 参数/工作区/配置无效、3 分析/存储/内部失败、4 协议或 schema 不兼容、130 取消。所有路径为工作区相对路径，check/impact/export 声明或等待目标 revision。
- (question) 配置所有权采用规则/忽略由仓库独占、运行偏好按 CLI>workspace>user>default 分层且安全护栏不可覆盖的拆分模型，还是对全部配置使用统一覆盖顺序，或只允许仓库级配置？
- (decision) 配置采用拆分所有权：.codegraph/rules.yaml 与 .codegraphignore 为仓库策略契约，不能被 VS Code 用户设置隐式覆盖；显式 CLI --config 只影响当前命令并必须报告来源。运行偏好优先级为 CLI > workspace settings > user settings > defaults。
- (decision) Workspace Trust、IPC 权限与令牌、路径越界、Webview CSP、默认不上传、禁止执行项目脚本/插件和服务端绝对安全预算属于不可覆盖硬护栏。doctor 必须显示配置生效值及来源。
