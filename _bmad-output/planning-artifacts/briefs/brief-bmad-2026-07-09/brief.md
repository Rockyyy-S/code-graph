# 产品简报：项目代码图谱

## 执行摘要

项目代码图谱是一款面向开发者和技术团队的 IDE 内代码结构感知与影响分析工具。它不是单纯把代码“画成图”，而是在开发者打开文件、修改代码、审查 PR 或接手陌生模块时，持续回答一个更实际的问题：我现在看的代码处在什么结构里，它依赖谁，谁依赖它，本次改动会不会破坏模块边界。

第一阶段建议聚焦 VS Code + TypeScript/JavaScript 中大型项目，采用本地优先架构：VS Code 插件和 CLI 作为入口，本地图谱服务负责索引、查询、规则检查和视图生成。MVP 先验证局部代码图谱是否能比目录树、搜索和静态文档更快帮助开发者判断结构影响。

这件事现在值得做，是因为 AI 编码和快速迭代正在放大结构漂移风险。代码可以更快生成，但团队更难持续确认模块边界、依赖方向和重构影响。项目代码图谱的机会在于把“结构理解”放回开发工作流现场，而不是让它停留在过期文档或一次性的可视化报告里。

## 问题

中大型代码库会随着时间偏离原本的结构意图。目录树只能说明文件放在哪里，搜索只能找到文本位置，文档经常滞后，PR diff 也很难展示一次改动对模块边界和依赖关系的影响。

典型痛点包括：

- 接手陌生代码时，不知道入口、模块职责和真实依赖链。
- 做重构时，不知道上游/下游影响范围，容易漏改或误改。
- 新增代码时，无意中引入跨层依赖、循环依赖或目录边界违规。
- PR 审查只能看到文件变更，难以看到结构变化和风险路径。
- AI 生成代码能运行，但可能破坏既有架构约束。

这些问题的共同成本是理解时间变长、改动风险升高、架构治理依赖人工记忆，团队越大越明显。

## 方案

项目代码图谱提供一个本地优先的代码结构感知层，把代码中的目录、文件、符号、导入导出、依赖关系和架构规则建成可查询图谱，并在 IDE 中按当前上下文展示相关子图。

核心体验是：

- 打开项目时，生成项目结构概览，展示目录/模块之间的真实依赖关系，而不只是文件夹层级。
- 打开文件时，自动展示当前文件邻域：它属于哪个模块，依赖谁，谁依赖它，是否处于循环依赖或边界风险中。
- 保存改动后，增量更新局部图谱，提示新增依赖边、循环依赖或规则违反。
- 审查 PR 时，输出结构变化摘要和影响路径。
- 后续通过 MCP / AI 上下文，把相关模块边界和依赖关系提供给 AI 编码工具。

第一版应优先服务“判断结构影响”，而不是追求全语言、全调用图或大规模全局可视化。

## 目标用户

主要用户是使用 VS Code 的 TypeScript/JavaScript 中大型项目开发者与团队：

- 日常开发者：需要快速理解当前文件的上下游影响。
- Tech Lead / 架构师：需要发现循环依赖、跨层依赖和架构漂移。
- 维护遗留系统的团队：需要降低重构和接手模块的风险。
- AI 编程重度团队：需要给 AI 提供结构化上下文和修改边界。

[ASSUMPTION] 第一阶段先验证个人或小团队的本地工作流价值，商业化包装暂不作为 MVP 的核心约束。

## 差异化

这个产品的差异化不在于“能画图”，而在于把结构感知嵌入开发现场：

- 相比普通代码地图：它强调当前文件聚焦、增量更新、规则检查和变更影响，而不是一次性大图。
- 相比搜索/代码导航：它展示模块、目录和文件之间的结构关系，而不只是定位定义或引用。
- 相比架构治理平台：它从轻量、本地、IDE 原生切入，先服务开发者日常判断，再扩展到团队规则和 CI。
- 相比 AI 编码助手：它提供可解释、可查询、可约束的代码结构上下文，帮助 AI 不越界修改。

产品语言应避免定位成“代码图谱可视化工具”，更准确的定位是“代码结构感知与影响分析工具”。

## MVP 范围

第一版目标是验证：开发者在打开或修改文件时，局部代码图谱能否显著降低结构理解和改动影响判断成本。

包含：

- VS Code 插件和 CLI。
- TypeScript/JavaScript 项目支持。
- 本地图谱索引：目录、文件、import/export、包依赖。
- 项目结构概览、当前文件邻域图、循环依赖视图、PR 变更影响摘要。
- 保存后增量更新，手动全量 rebuild。
- 基础规则：循环依赖、禁止目录引用、层级依赖约束。
- 本地存储和查询服务，源码与图谱默认不上传。

明确不做：

- 不做全语言支持。
- 不做精准运行时调用图。
- 不做默认全量大图展示。
- 不做云端团队协作。
- 不做 AI 自动重构。
- 不把重型图数据库、跨语言 SCIP 全量索引或 CPG 数据流分析放进 MVP 主路径。

## 技术边界

MVP 技术栈保持轻量，但从第一天预留升级边界：

- VS Code 插件只做薄客户端，不承担重索引和复杂布局。
- 索引器、图查询、规则引擎放在独立本地图谱服务中。
- 存储通过 `GraphStore` 接口访问，SQLite 只是第一种实现。
- 分析能力通过 `AnalyzerAdapter` 接入 Tree-sitter、LSP，未来可接 SCIP / CPG。
- 可视化层只消费 `GraphViewModel`，不把 Cytoscape.js JSON 当持久图谱模型。
- 节点和边使用稳定 ID，并记录来源、置信度、版本和更新时间。

这些边界的目的不是提前复杂化，而是避免 MVP 被 SQLite、Cytoscape.js 或 VS Code Webview 锁死。

## 成功标准

MVP 可用以下指标判断是否值得继续投入：

- 打开项目后 60 秒内生成第一版结构概览。
- 打开文件后 300ms 内显示缓存图谱，并在后台刷新。
- 保存文件后 2 秒内更新局部依赖。
- 能准确识别 80% 以上 import/export 依赖。
- 能发现循环依赖和配置的跨层规则违反。
- 用户能在 3 分钟内回答“改这个文件会影响哪里”。
- [ASSUMPTION] 早期用户反馈中，“理解结构更快”比“图更好看”更重要。

## 后续愿景

如果 MVP 验证成立，项目代码图谱可以从本地开发辅助逐步扩展为团队级结构治理层：

1. 加入 `.codegraph/rules.yaml`、CI 命令和 PR 影响分析，让结构规则进入团队流程。
2. 增加依赖矩阵、分层图、历史趋势和热点模块分析，服务重构与架构治理。
3. 暴露 MCP server，把当前文件上下文、依赖邻域、规则检查和变更影响提供给 AI 工具。
4. 在团队共享、跨仓库分析和安全/合规场景成熟后，再考虑云端或私有部署。

长期愿景不是成为“另一个代码浏览器”，而是成为开发者、架构师和 AI agent 共用的代码结构事实层。

## 关键未决问题

- 产品首批试用对象是个人开发者、小团队，还是企业内部技术团队？
- 第一版是否必须支持 PR 工作流，还是先专注 IDE 内打开/保存场景？
- 架构规则配置应走最小 YAML，还是提供可视化规则编辑？
- 是否需要从第一版提供 MCP 接口，还是作为验证后的第二阶段能力？
- 早期成功更看重开发者体验指标，还是团队架构治理指标？
